# Achievement Overlay

## Overview

Create a standalone C# app that monitors GBE/GSE emulator achievement unlocks and displays Steam-like overlay notifications over the game window. Inspired by [Achievement Watchdog](https://github.com/50t0r25/achievement-watchdog) (Python, Windows toast notifications), but uses a custom transparent popup for a more in-game feel. Zero interaction with the game process — purely filesystem-based detection, which means no game crashes from library manipulation.

## Context

- Files involved:
  - This project root — new C# project (entire app)
- Related patterns:
  - `../achievement-watchdog/achievement_watchdog.py` — existing Python implementation with the same detection logic (filesystem watching, JSON cross-referencing, notification display). We follow the same approach for achievement detection.
  - [`url-cleaner`](../../../../url-cleaner/) — existing C# tray app with the same lifecycle pattern (system tray icon, context menu with Pause/Start with Windows/Open Config/Exit, settings persistence, registry auto-start). We adopt the same structure and patterns.
- Dependencies: .NET 10.0 SDK (matching url-cleaner), Windows Forms (for `NotifyIcon` tray — matching url-cleaner), WPF (for the transparent overlay window)

**Why C#**: Native Windows APIs for `FileSystemWatcher`, transparent windows, and system tray — no external dependencies needed for core functionality. Produces a small standalone exe without a runtime dependency (unlike Python/PyInstaller). Matches the url-cleaner project's stack, enabling pattern reuse (tray lifecycle, settings, auto-start).

**Why WinForms tray + WPF overlay window (hybrid)**: The url-cleaner pattern uses WinForms `NotifyIcon` for the tray (simple, proven, no NuGet dependency). But WinForms can't do transparent borderless windows with animations — WPF can. The app hosts both: WinForms `ApplicationContext` for the tray lifecycle (reusing url-cleaner's pattern), WPF `Window` for the overlay popup. This is a standard .NET pattern — both frameworks coexist in the same process.

## Design Notes

**Data flow**: `FileSystemWatcher` detects `achievements.json` change in `%appdata%/GSE Saves/<appid>/` -> check file modification time (skip if unchanged) -> read file, diff against cached state to find new unlocks -> look up `<appid>` in game cache (maps appid to game directory via `steam_appid.txt` scanning) -> read `steam_settings/achievements.json` for display name, description, icon path -> show overlay notification.

**Two JSON formats involved**:
- GSE Saves `achievements.json` (local unlock state): `{"ACH01": {"earned": true, "earned_time": 1774855788}, ...}` — keys are achievement internal names, values have earned bool and unix timestamp.
- Game `steam_settings/achievements.json` (metadata schema, generated by [gbe_fork_tools/generate_emu_config](https://github.com/Detanup01/gbe_fork_tools/tree/main/generate_emu_config_old)): array of objects with `name`, `displayName` (string or multi-language object), `description` (string or multi-language object), `icon`, `icon_gray`, `hidden`. Display text fields can be either a plain string or an object like `{"english": "...", "german": "..."}`.

**Game cache**: On startup, scan all configured game paths recursively for `steam_appid.txt` files. Read the appid from each, find the corresponding `steam_settings/achievements.json` in the same directory, and cache the mapping `appid -> achievements metadata path`. When a new appid folder appears in GSE Saves that isn't in the cache, re-scan to pick it up (same as Achievement Watchdog does). Log all detected games and cache state to a log file.

**Logging**: Write a log file (`achievement-overlay.log` beside the exe) with key runtime details: detected games on startup, cache hits/misses, achievement unlock events, notification dispatch, errors. Clear on each launch (same as Achievement Watchdog). Use simple file logging — no logging framework needed.

**Overlay window**: WPF borderless, transparent (`AllowsTransparency=true`, `WindowStyle=None`, `Background=Transparent`), `Topmost=true`. Positioned at bottom-right of the foreground window at the moment the notification fires (no real-time tracking). Dark semi-transparent rounded rectangle with achievement icon on the left, achievement name in bold, description below in smaller text. Window and icon size proportional to the game window (~15% width, icon scaled accordingly) rather than fixed pixel size — handles different resolutions and window sizes. Slide-up animation on appear (translate Y, ~300ms ease-out), hold for 5 seconds, fade-out (~500ms), then close.

**Notification queue**: If multiple achievements unlock simultaneously (batch write to achievements.json), queue them and show one at a time with a short gap between.

**Configuration** (`settings.json` beside the exe):
```json
{
  "gseSavesPath": "%appdata%\\GSE Saves",
  "gamesPaths": "C:\\Games;D:\\Games",
  "language": "english",
  "soundEnabled": true,
  "soundPath": ""
}
```
- `gseSavesPath` — defaults to `%appdata%\GSE Saves`, customizable
- `gamesPaths` — semicolon-separated list of game directories to scan
- `language` — preferred language for display text, fallback to english
- `soundEnabled` — toggle notification sound (persisted, also togglable from tray)
- `soundPath` — custom sound file path; empty string or missing = use embedded default

Environment variables like `%appdata%` are expanded at runtime. Missing file -> create with defaults. Settings are hot-reloaded on file change (same pattern as url-cleaner: check `GetLastWriteTimeUtc` before each use).

**Sound**: Bundle a default unlock sound as an embedded resource. If `soundPath` is set in config and the file exists, use that instead. Play via `System.Media.SoundPlayer` (.wav) or NAudio (if mp3 support needed). Fire-and-forget, non-blocking.

**System tray** (matching url-cleaner's `TrayApplicationContext` pattern): WinForms `NotifyIcon` with context menu:
- **Sound Enabled** — checkbox, persists to `settings.json` via `UpdateConfigValue`
- **Pause Notifications** — checkbox, suppresses popups while checked (not persisted — resets on restart)
- **Start with Windows** — checkbox, writes/deletes registry key at `HKCU\SOFTWARE\Microsoft\Windows\CurrentVersion\Run` (same as url-cleaner)
- **Open Config Location** — opens Explorer with `settings.json` selected (same as url-cleaner)
- **Exit** — disposes watcher, stops queue, exits

**Project structure** (matching url-cleaner's flat layout — no `Services/` subfolder, since there are few enough files):
```
achievement-overlay.slnx
config/
  default.json              — embedded default settings
src/
  AchievementOverlay.csproj
  Program.cs                — entry point, WinForms app context init
  TrayApplicationContext.cs — tray icon, context menu, lifecycle
  NotificationWindow.xaml/.cs — WPF overlay popup window
  AchievementWatcher.cs     — FileSystemWatcher + change detection
  GameCache.cs              — scans game paths, maps appid -> metadata
  AchievementMetadata.cs    — JSON models + display text resolution
  NotificationQueue.cs      — queues and dispatches overlay popups
  SoundPlayer.cs            — plays unlock sound
  AppConfig.cs              — loads/saves settings.json, hot-reload
  Assets/
    achievement_sound.wav   — default unlock sound (embedded resource)
    icon.ico                — tray icon
tests/
  AchievementOverlay.Tests.csproj
  AchievementWatcherTests.cs
  GameCacheTests.cs
  AchievementMetadataTests.cs
  NotificationQueueTests.cs
  AppConfigTests.cs
```

This mirrors url-cleaner's flat `src/` layout (no nested folders for a small app), `config/default.json` as embedded resource, `tests/` as a sibling directory, and `.slnx` solution file format.
